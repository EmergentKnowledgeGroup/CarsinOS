# CarsinOS ExecAss Frontend Experience Brief

## Audience

Claude, as the frontend experience and visual design owner.

This is a creative brief, not a prescribed layout or implementation plan. You own the art direction, information design, interaction design, motion, responsiveness, component composition, and visual personality.

The boundaries below protect the product idea. Inside them, take creative ownership.

## The Assignment

Redesign CarsinOS so it feels like working with a highly capable executive assistant that happens to command a mission-control system.

It must stop feeling like the user has been seated in front of a spaceship console they have never seen before.

The current frontend exposes too much organizational and operational machinery at the same level. It asks the user to learn the product's topology before receiving its value. The redesign should reverse that relationship.

## North Star

> Mission control in capability. Executive assistant in experience.

The user should be able to open CarsinOS, understand the situation, delegate an outcome, answer the few decisions that genuinely require them, and leave again.

The interface should feel calm, capable, personal, trustworthy, and easy to check in on. It should not feel like an admin portal, infrastructure monitor, project-management maze, or shallow chatbot.

## The User Relationship

ExecAss is the user's primary relationship with CarsinOS.

The user speaks in outcomes. ExecAss decides how the organization should act. The system shows enough context for confidence and offers deeper truth on demand.

The user is the principal, not the dispatcher. ExecAss is the orchestrator, not merely another agent tile or chat tab.

## The Everyday Product Shape

The daily experience should resolve into three understandable places:

### ExecAss

The place to understand what matters, delegate work, make decisions, and review outcomes.

### Work

The place to inspect and shape durable work: plans, execution, routines, and history.

### Inbox

The place for human attention: decisions, conversations, replies, and recovery items.

Setup remains available as a utility space for configuration, connections, team administration, permissions, and other infrequent technical work.

Specialist, forensic, and operator views remain reachable through context and advanced access. They should not all compete for primary navigation attention.

## ExecAss Home: Required Workable Spaces

The composition is yours, but the experience needs room for these concepts:

### Briefing

A short, intelligent summary of what changed, what matters now, and whether anything requires the user.

It should read like an assistant's briefing, not a row of telemetry cards.

### Ask or Delegate

A clear, inviting place to give ExecAss an outcome in ordinary language.

The user should not need to choose whether the request belongs to a board, agent, job, schedule, connector, room, or runbook before submitting it.

### Needs You

The most important attention space. It contains only decisions, approvals, clarifications, replies, or recovery choices that genuinely need the user.

Each item should explain the decision in human terms and make the recommended next action obvious.

### In Motion

A calm view of what ExecAss and the organization are handling now. It should communicate confidence and progress without requiring constant supervision.

### Done Since You Checked

Meaningful outcomes, not a firehose of completed internal steps. Each outcome should offer a direct route to its receipt or evidence.

### Next

Upcoming commitments, routines, deadlines, or follow-ups the user should know about, without turning the page into a calendar dashboard.

### Ask About Anything

The user should be able to question a summary, request evidence, change direction, stop work, or drill deeper naturally from the context in front of them.

## Work: Preserve Power Without Preserving Clutter

The Work area should give the current durable work surfaces a comprehensible home:

- **Plan:** intent, goals, strategy, sequencing, and major choices.
- **Execution:** active work, ownership, progress, dependencies, and exceptions.
- **Routines:** scheduled and recurring work.
- **History:** completed work, runbooks, receipts, and reviewable evidence.

Existing boards, strategy, calendar, runbook, team workload, and related capabilities should be preserved within this clearer mental model. They do not all need equal prominence at all times.

## Inbox: Human Attention, Not System Activity

The Inbox should collect what crosses the human boundary:

- **Decisions:** approvals, choices, and clarifications.
- **Conversations:** Agent Mail, rooms, external replies, and other correspondence.
- **Recovery:** failed or blocked work where the user has a meaningful option.

Routine system activity does not belong here merely because it generated an event.

## Approval Experience

An approval should feel like ExecAss bringing the user a considered decision, not a security system presenting a raw action payload.

The experience should communicate:

- What ExecAss recommends.
- Why the decision is needed now.
- What will happen if approved.
- The meaningful impact or risk.
- The scope of permission being requested.
- The available alternatives: approve and continue, revise, decline, or stop.

After approval, the original work should visibly continue. The user should not be asked to perform a second, hidden resume step.

Technical identifiers and exact authorization evidence should remain inspectable, but they should not dominate the decision surface.

## Information Hierarchy

The default interface should answer these questions in order:

1. Does anything need me?
2. What is ExecAss handling?
3. What changed or finished?
4. What is coming next?
5. How do I delegate or redirect something?
6. Where is the proof or deeper control if I want it?

System health, schedulers, job counts, breaker states, provider details, model routing, agent identifiers, and event streams matter. They should surface prominently when something is wrong or when the user deliberately opens advanced detail, not remain permanently equal to the user's goals.

## Progressive Disclosure and Trust

The experience should be simple at first glance and rigorous one layer down.

Every important summary, recommendation, and completion should lead cleanly to its underlying task, conversation, artifact, runbook, board, event history, or technical detail. This preserves trust without making the user live inside the audit trail.

Do not solve complexity by hiding truth. Solve it by presenting the right truth at the right depth.

## Reef

Reef is a strong piece of CarsinOS identity. Preserve and develop it as an optional “watch the team” or ambient organization view.

It can provide charm, presence, coarse activity, collaboration cues, and a feeling that the organization is alive. It should remain grounded in truthful, privacy-preserving state and offer paths into real work.

Reef should not become a mandatory control scheme or the only way to understand what needs attention. It is an expressive lens on ExecAss and the team, not a replacement for clear language and direct action.

## Tone and Feel

Aim for:

- Calm authority rather than dashboard urgency.
- Personal assistance rather than system administration.
- Glanceable understanding rather than dense monitoring.
- Confident recommendations rather than unexplained prompts.
- Visible progress rather than constant activity noise.
- Warmth and identity without sacrificing precision.
- Serious control without visual intimidation.

When nothing needs the user, the interface should feel reassuringly quiet.

## Mobile and Small-Screen Intent

On smaller screens, protect the assistant relationship above all else:

- Needs You must remain obvious and plainly labeled.
- Delegation must remain immediate.
- Decision actions must be easy to understand and use one-handed.
- Active work should summarize before it expands.
- Advanced controls should remain reachable without crowding the daily experience.

Do not reduce the product to unlabeled icons or hide critical human-attention state behind a menu.

## Functional Capabilities That Must Survive

The redesign must retain access to the current product's substantive power:

- ExecAss conversation and delegation.
- Focus and attention management.
- Boards, plans, execution, and strategy.
- Calendar, schedules, jobs, and recurring work.
- Runbooks, history, events, and evidence.
- Agent Mail, rooms, teams, workload, and collaboration.
- Approvals, recovery, incidents, and stop controls.
- Connectors, memory, models, providers, authentication, and configuration.
- Cockpit and other advanced operational views.
- Reef as an optional visual organization view.

The challenge is not to remove functionality. It is to stop presenting every capability as an equal daily responsibility.

## Creative Freedom

You are explicitly free to determine:

- The visual language and art direction.
- The layout and composition of each space.
- The navigation expression.
- How summaries, cards, conversations, timelines, or spatial views behave.
- Motion, transitions, responsive transformations, and ambient states.
- How Reef connects aesthetically and interactively to the core experience.
- How personality is expressed without weakening clarity.

You do not need to preserve the current screen composition, tab structure, dashboard card language, or visual hierarchy.

You do need to preserve the product concepts, safety boundaries, deep access, and truthful state described here.

## Anti-Goals

- Do not produce a prettier operator dashboard.
- Do not turn ExecAss into a generic chat window with the real product hidden elsewhere.
- Do not make users learn CarsinOS's architecture before they can delegate work.
- Do not surface raw IDs, provider settings, routing, or telemetry as the opening experience.
- Do not treat every event as a notification or every internal action as user-facing progress.
- Do not weaken approvals or conceal consequential actions.
- Do not delete advanced controls merely because they are not primary.
- Do not hide failures behind reassuring language.
- Do not make Reef the only understandable navigation model.

## Backend Dependency

The frontend redesign depends on backend behavior that supports:

- One continuous task lifecycle.
- Clear waiting-for-user state.
- Approval that continues the original work.
- Useful scoped authority.
- One ExecAss intake for outcomes.
- Trustworthy Needs You, In Motion, Done, Next, and Receipts state.
- Deep links to authoritative evidence and controls.

The frontend can explore the experience before all backend work lands, but it should not invent fake continuity to compensate for missing behavior.

## Experience Success Tests

The redesign is successful when a user can:

- Open CarsinOS and understand the important situation in roughly ten seconds.
- Tell immediately whether anything needs them.
- Delegate an outcome without selecting an internal subsystem.
- Leave ordinary work to ExecAss without continuous monitoring.
- Resolve a consequential decision with enough context to feel informed.
- See the original work continue after approval.
- Understand what completed in outcome language.
- Reach the evidence or advanced control for any important claim in one deliberate step.
- Use the product comfortably on desktop and mobile.
- Access every important current capability without having all of it permanently in their face.

## Final Design Test

Ask one question whenever a major design decision is made:

> Does this help the user work with ExecAss, or does it ask the user to operate CarsinOS?

If it primarily teaches the user to manage the machinery, redesign it. If it helps the user delegate, decide, understand, and trust while preserving access to the machinery, it belongs.
