# Assistant Tool Contract v1 Spec (CarsinOS)

Spec-only: this document defines the contract and acceptance criteria. It does not implement runtime changes.

Contract version: `assistant.tools.v1`

## Purpose

Give the "main assistant" a **first-class, tool-call-native** way to orchestrate other workers (sub-agents) without relying on prompt hacks or freeform JSON protocols.

This contract is designed so:

- The assistant can "hire" permanent employees (semi-permanent identities).
- The assistant can create fast throwaway contractors (session-only helpers).
- Workers can run on **different providers/models than the main assistant**.
- High-risk actions are **approval-gated** and **auto-finished** by CarsinOS after approval.
- Guardrails remain enforced (auth/roles/approvals/audit/budgets/breakers/lane locks).
- NumquamOblita (NO) is **not** directly called by the assistant in v1, but future tenancy fields are reserved now.

## Non-Goals

- No bypass of CarsinOS policy engine / guardrails.
- No per-worker tool sandboxing in v1 (workers share the global tool policy + approvals).
- No direct assistant-to-NO execution path in v1.
- No UI/UX changes are required to implement this contract (operator surfaces may still improve later).

## Glossary (Caveman Version)

- **Agent**: a saved identity in the system ("employee record"). Has an `agent_id` and a name.
- **Session**: a chat/workspace thread for an agent ("their inbox"). Has `session_id`.
- **Run**: one execution attempt inside a session ("the job run"). Has `run_id`.
- **Worker**: what the assistant talks about.
  - **Employee worker**: `agent + session` (semi-permanent identity).
  - **Contract worker**: `session only` (no dedicated agent; shares the boss agent identity under the hood).
- **Main assistant**: the boss worker, determined by the root session (`root_session_id`).

## Locked Decisions (Updated)

1. Transport mode: Both (MCP tools + function-calling mirror).
2. Canonical protocol: MCP-style tools; function-calling mirror reuses the same schemas.
3. v1 spawn scope: Full spawn (worker lifecycle + assignment + join + close).
4. Spawn authority: Main assistant only (per root session).
5. Spawn limits: Moderate cap defaults, config-driven.
6. Safety: Guardrails stay on (existing budgets/breakers/approvals/locks).
7. Worker lifecycle: Hybrid (employees + contractors).
8. Permanent employees must be approval-gated: Always.
9. Employee "brain" updates (provider/model) are approval-gated: Always.
10. Workers may run different provider/model than the main assistant: Supported by design.
11. Auth profile selection for runs: System chooses healthiest eligible by default.
12. Direct assistant-to-NO: Deferred; tenancy fields reserved now.

## System Model (Ownership, Identity, and Roster)

### Ownership Key (Boss Key)

Workers are scoped to the main assistant. The system derives a stable "boss key" for the roster:

1. Load root session by `root_session_id`.
2. Let `boss_agent_id = root_session.agent_id`.
3. Let `boss_key = boss_agent_id` (primary) and `root_session_id` (secondary).

Rules:

- A worker can only be listed/modified by tool calls for the same `boss_key`.
- If `root_session.agent_id` is missing, employee hiring is forbidden (fail closed). Contractor sessions may still be allowed, but MUST remain scoped to `root_session_id`.

### Worker Identifiers

- `worker_key`: a human-friendly stable identifier chosen by the assistant (required, unique per boss).
  - Pattern: `^[a-z0-9][a-z0-9_-]{0,63}$`
- `agent_id`: generated by the system for employee workers (assistant never chooses it).
- `session_id`: created by the system.

## Worker Types v1

### Employee Worker (Permanent/Semi-Permanent)

Employee worker maps to:

- `agent_id` (system-generated)
- 1 primary `session_id` (default "inbox")
- Optional additional per-task sessions (fresh inbox per assignment) when explicitly requested by the assistant

Employee worker is used for "hire a specialist" patterns:

- Researcher on a cheaper model
- Archivist/document-writer on a consistent model
- Any other durable specialist

### Contractor Worker (Session-Only)

Contractor worker maps to:

- a new `session_id` only
- no dedicated `agent_id` (under the hood, session uses the boss agent id)

Contractor worker is used for "quick helper" patterns:

- One-off summarization
- Short task decomposition

Contractor rule (v1):

- Contractors are immutable. If you want "a different contractor", spawn a new one and archive the old one.

## Templates and Overrides (Brains and Job Descriptions)

Workers are created from a template, plus optional overrides. Templates define defaults for:

- `display_name`
- `instructions` (job description/system hint)
- `run_defaults.model_provider`
- `run_defaults.model_id`

The assistant may override any of these at hire-time (and for employees, at update-time with approval).

Default built-in templates (v1 keys):

- `general`
- `researcher`
- `archivist`

Implementation note: templates MUST be config-driven (not hardcoded) long-term; `assistant.capabilities.get` exposes what templates exist.

## Approval Model (Mandatory for Hiring + Brain Updates)

### What Requires Approval

- Hiring an employee worker (`assistant.worker.spawn` with `worker_kind=employee`)
- Updating an employee's provider/model (`assistant.worker.update` when changing brain fields)

### Auto-Finish Rule (No Extra Assistant Step)

When an approval is granted:

- CarsinOS completes the requested action under a privileged internal actor (service internal).
- The assistant does NOT need to re-issue the tool call.
- The assistant learns completion by polling:
  - `assistant.worker.list` / `assistant.worker.status`
  - `assistant.approvals.list`

### How a Tool Reports an Approval Block

The tool response returns:

- `status = "blocked"`
- `reason_code = "APPROVAL_REQUIRED"`
- `data.approval_id` plus a short operator-facing summary

## Transport and Parity Rules

### Canonical MCP Bridge

- Endpoint: `POST /api/v1/assistant-tools/mcp`
- Uses JSON-RPC shape compatible with existing MCP patterns (`initialize`, `tools/list`, `tools/call`).

### Function-Calling Mirror

- Endpoint: `POST /api/v1/assistant-tools/call`
- Takes `{ tool_name, arguments }` and returns the same normalized result shape.

### Parity Rule

MCP and function-calling are required to be **semantically equivalent**:

- Same tool names
- Same required/optional fields and constraints
- Same meaning of fields and reason codes
- Byte-for-byte equality is NOT required (field order/whitespace/transport metadata may differ)

## Common Envelope (Normalized)

The system normalizes all transports to the following request/response envelope. The assistant is not required to supply identity fields; CarsinOS fills them from the current run/session context.

### Request Envelope

- `contract_version` (required): `assistant.tools.v1`
- `request_id` (required): caller-supplied UUID or request token
- `root_session_id` (required)
- `root_run_id` (optional)
- `caller_agent_id` (required): the main assistant agent id (derived; fail closed if mismatch)
- `memory_scope` (optional, reserved for future NO direct routing). Shape:
  - `tenant_key` (string)
  - `namespace` (string)
  - `db_hint` (string)
- `tool_name` (required)
- `arguments` (required object)

### Response Envelope

- `contract_version`
- `request_id`
- `status`: `ok | error | blocked`
- `reason_code` (optional, stable enum)
- `message` (short operator-facing text)
- `data` (tool-specific object)
- `audit_ref` (security audit id, if available)
- `timing_ms`

## Auth, Roles, and Guardrails (v1)

- All assistant-tools endpoints require bearer auth (no anonymous access).
- Role policy (default):
  - Read-only calls (`assistant.capabilities.get`, `assistant.worker.list`, `assistant.worker.status`): `operator_admin | operator_readonly | automation_runner`
  - Worker mutations (`assistant.worker.spawn`, `assistant.worker.update`, `assistant.worker.assign`, `assistant.worker.join`, `assistant.worker.close`): `operator_admin | automation_runner`
  - Approvals listing (`assistant.approvals.list`): `operator_admin | operator_readonly`
  - Mail (`assistant.mail.*`): `operator_admin | operator_readonly | automation_runner` plus thread membership enforcement
- Spawn authority rule:
  - Only the main assistant identity for the root session may create/list/modify workers for that boss key.
  - If `caller_agent_id` does not match `root_session.agent_id`, fail closed with `AUTH_FORBIDDEN` or `POLICY_DENY`.
- Guardrails rule:
  - Existing CarsinOS run/job/tool guardrails remain active (budgets, breakers, lane locks, approvals, auditing).

## Reason Codes (Stable)

Base codes (shared):

- `INVALID_INPUT` (caller fix required)
- `AUTH_REQUIRED` (missing/invalid token)
- `AUTH_FORBIDDEN` (role/policy deny)
- `POLICY_DENY` (guardrail deny; not retryable without changes)
- `APPROVAL_REQUIRED` (blocked pending human approval)
- `RATE_LIMITED` (retryable with backoff)
- `RUN_LANE_CONFLICT` (retryable later)
- `BUDGET_EXCEEDED` (terminal for that run)
- `BREAKER_OPEN` (retryable after cooldown)
- `DEPENDENCY_UNAVAILABLE` (retryable)
- `TIMEOUT` (retryable)
- `INTERNAL_ERROR` (retryable with caution)

Worker-specific codes:

- `WORKER_NOT_FOUND`
- `WORKER_ARCHIVED`
- `WORKER_PENDING_APPROVAL`

```json
{
  "type": "object",
  "additionalProperties": false,
  "required": ["worker_key", "approval_id", "requested_action"],
  "properties": {
    "worker_key": { "$ref": "#/$defs/worker_key" },
    "approval_id": { "type": "string" },

    "summary": { "type": "string" }

```

Rules:

- If `worker_kind=employee`: MUST return `blocked` with approval unless caller is already a privileged operator context and policy allows auto-approval (default: approval required).
- If `worker_kind=contract`: SHOULD be `ok` unless caps/policy deny.
- `session_mode` is the worker's default assignment behavior:
  - `persistent`: use the worker's primary session for assignments (keeps context).
  - `fresh_per_task`: create a new session for each assignment (clean inbox per job).
  - The assistant may override per task via `assistant.worker.assign.session_mode`.

Rules:

- Creates a user message in the worker session with `task`.

  - For employee workers:
    - `persistent`: use the worker's primary session.
    - `fresh_per_task`: create a new per-task session for that employee and run there.
  - For contractor workers: always run in the contractor session; if `fresh_per_task` is requested, fail with `INVALID_INPUT`.

### Example 1: Hire a Research Employee (Approval-Gated)

1. Assistant calls `assistant.worker.spawn` with `worker_kind=employee`, `template_key=researcher`.
2. Tool returns `blocked` with `approval_id`.
3. Human approves in Mission Control.
4. CarsinOS auto-creates the agent+session and marks the worker active.
5. Assistant polls `assistant.worker.status` until `status=active`.
6. Assistant calls `assistant.worker.assign` with a task.

### Example 2: Use a Contractor for a One-Off Task

1. Assistant calls `assistant.worker.spawn` with `worker_kind=contract`.
2. Tool returns `ok` with `session_id`.
3. Assistant calls `assistant.worker.assign`.
4. Assistant calls `assistant.worker.join` until `completed`.
5. Assistant calls `assistant.worker.close` to archive.
